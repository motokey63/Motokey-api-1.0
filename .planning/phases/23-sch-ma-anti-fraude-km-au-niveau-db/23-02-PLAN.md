Créer `scripts/test-releves-km-trigger.js`, un script Node autonome utilisant `pg` (require dotenv + Client, exactement comme bootstrap-fresh-schema.js), connecté via `FRESH_DB_URL`. Reprendre le garde-fou anti-prod (refuser si la string contient `rzbqbaccjyxvtlnfitrr`) et ne jamais logger la string. Le script suppose que le schéma (schema.sql, donc les tables/triggers de 23-01) est déjà bootstrappé sur la DB cible (le plan 23-04 lance d'abord bootstrap-fresh-schema.js puis ce script).

Le script insère ses propres fixtures (un garage, une moto avec `motos.km` pré-peuplé) puis exécute et ASSERTE ces cas, chacun affichant `PASS <case>` / `FAIL <case>` et incrémentant des compteurs (style OK/KO de test-api.js) ; exit code 1 si au moins un FAIL :

1. `accept` — moto.km=10000, INSERT releves_km (km=12000, type_evenement='lecture'). Assert : 1 ligne dans releves_km ET motos.km == 12000 (sync AFTER INSERT).
2. `reject-regression` — après le cas 1 (max=12000), INSERT releves_km (km=9000, 'lecture'). Assert : l'insert est annulé (0 ligne renvoyée : `INSERT ... RETURNING id` renvoie 0 row) ET une ligne existe dans releves_km_rejets avec km_tente=9000 et km_actuel=12000 ET motos.km reste 12000.
3. `null-safe-baseline` — NOUVELLE moto avec motos.km=40000 et ZÉRO ligne releves_km. INSERT releves_km (km=100, 'lecture'). Assert : REJET (annulé + ligne de rejet km_actuel=40000), motos.km reste 40000. (C'est l'état par défaut de toute moto prod le jour du déploiement — pas un edge case.)
4. `counter-replacement-bypass` — moto avec relevés jusqu'à km=40000. INSERT releves_km (km=5, type_evenement='remplacement_compteur'). Assert : ACCEPTÉ (bypass), motos.km devient 5. Puis INSERT (km=100,'lecture') → ACCEPTÉ (chaîne fraîche, pas comparé aux 40000 d'avant). Puis INSERT (km=50,'lecture') → REJETÉ (< 100 post-reset). Prouve "démarre une chaîne monotone fraîche".
5. `conso-check-violation` — INSERT consommables (type_consommable='invalide') → assert erreur Postgres code `23514` (check_violation). Puis INSERT chacun des 9 types valides → assert succès.

Le script nettoie ses fixtures en fin d'exécution (DELETE des lignes créées) ou opère dans une transaction rollback si plus simple. Utiliser des UUID de fixtures explicites pour un nettoyage ciblé.

IMPORTANT : ce script est écrit AVANT que le trigger existe côté DB cible. Au moment de sa création (23-02) il n'est PAS exécuté (FRESH_DB_URL peut être absent / DB pas encore bootstrappée) — seul `node --check` valide sa syntaxe ici. Son exécution verte est le gate de 23-04.
