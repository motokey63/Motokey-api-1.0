    ```sql
    -- Migration 16 : Table client_device_tokens — jetons push mobile (Expo)
    -- À appliquer manuellement via Supabase Dashboard > SQL Editor

    CREATE TABLE client_device_tokens (
      id           UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
      client_id    UUID        NOT NULL REFERENCES clients(id) ON DELETE CASCADE,
      token        TEXT        NOT NULL UNIQUE,
      platform     TEXT        NOT NULL CHECK (platform IN ('ios', 'android')),
      last_used_at TIMESTAMPTZ NOT NULL DEFAULT now(),
      created_at   TIMESTAMPTZ NOT NULL DEFAULT now()
    );

    CREATE INDEX idx_client_device_tokens_client ON client_device_tokens(client_id);

    COMMENT ON TABLE client_device_tokens IS 'Jetons push Expo (mobile) liés à un client. Un client peut avoir plusieurs devices actifs (multi-appareil, D-01). UNIQUE(token) permet la réassignation upsert (D-02) si un device change de propriétaire.';
    ```

    Design constraints that MUST be honored (do NOT change these):
    - Key on `client_id` (FK to `clients(id)`), NOT `auth_user_id` — per CONTEXT.md D-01. (An earlier milestone-level research pass proposed auth_user_id; it is superseded by D-01.)
    - `UNIQUE` on `token` ALONE (not a composite key) — this is what makes D-02's upsert `onConflict:'token'` reassign a device to a new client on re-submit.
    - `ON DELETE CASCADE` so tokens vanish if a client is deleted.
    - `platform` is required with a CHECK IN ('ios','android') — mirrors the garage_users.role CHECK style in migration 12.
    - No RLS policy is added: `SBLayer.supabase` is the service-role client and bypasses RLS; no anon/authenticated path queries this table.
  </action>
