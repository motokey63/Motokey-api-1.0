    Add a silent refetch-on-focus to both tab screens, guarded so it does NOT double-fetch
    on the initial mount (the existing mount `useEffect` already owns the first load, with
    its spinner). Leave every existing effect, load(), pull-to-refresh, and loading path
    exactly as-is — this is an additive change only.

    In BOTH files:
    1. Import `useFocusEffect` from `expo-router`.
       - motos/index.tsx: extend the existing `import { router, Stack } from 'expo-router';`
         to `import { router, Stack, useFocusEffect } from 'expo-router';`
       - devis/index.tsx: add a new line `import { useFocusEffect } from 'expo-router';`
       - Ensure `useRef` is imported from 'react' (add it to the existing react import).
    2. Add a first-focus guard ref inside the component:
         `const isFirstFocus = useRef(true);`
    3. Add the focus effect AFTER the existing mount `useEffect`:
         ```
         useFocusEffect(
           useCallback(() => {
             if (isFirstFocus.current) {
               // Initial focus coincides with mount — the mount effect already loaded.
               isFirstFocus.current = false;
               return;
             }
             // Silent background refresh on tab-return (no spinner, list stays visible).
             load();
           }, [load])
         );
         ```
       `useCallback` is already imported in both files. Do NOT wrap this in setLoading(true)
       — a focus-return must refresh in the background without flashing the full-screen
       spinner or hiding the current list.

    Why the ref guard (not a pure useFocusEffect replacing the mount effect): the initial
    focus fires simultaneously with mount, so replacing the mount effect risks a spinner
    regression on Devis (whose loading toggle lives in the mount wrapper, not in load()).
    Skipping the first focus keeps the initial-load path 100% untouched and adds only the
    subsequent-focus refresh — the minimal fix for this narrow staleness bug. This mirrors
    the "guard against redundant fetch" discipline already used elsewhere (AuthContext's
    single-flight refresher, STATE.md Phase 14-02 decision) without adding new machinery.
  </action>
